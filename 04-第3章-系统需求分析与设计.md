第 3 章 系统需求分析与设计

3.1 可行性分析

家政服务管理系统面向用户、服务人员与平台运营多主体协同，目标是将服务分类、人员、订单、支付与评价等数据统一纳入平台治理，形成可追踪的业务闭环。项目已具备功能模块清单、数据库表字段说明与运行清单，业务边界清晰，需求可落地。结合现有开源技术栈与本地部署条件，系统建设在目标、资源与实施路径上具备可行性。

3.1.1 技术可行性

本系统采用 Spring Boot 3.2 与 Java 17 搭建后端，配合 MyBatis-Plus 提供稳定的 ORM 与分页查询能力，接口通过 REST 形式对前端开放。前端使用 Vue 3、Element Plus 与 ECharts 组织页面交互与数据可视化，登录鉴权依托 Spring Security 与 JWT，支付与退款流程可由 Alipay SDK 与 WebSocket 通知支撑。整体技术路线成熟，开发与部署风险可控，满足家政服务高频业务的并发与扩展要求。

3.1.2 经济可行性

系统核心组件均为开源框架，数据库采用 MySQL，避免商业授权成本；前后端分离架构可复用现有服务器与运维工具，部署成本主要集中在基础硬件与人员配置。通过统一接口与模块化设计减少二次开发工作量，运行与维护以常规服务器、备份与日志即可支撑，长期投入可控，具备经济可行性。

3.1.3 操作可行性

系统面向用户、服务人员与管理员三类角色提供清晰操作入口，服务浏览、预约下单、支付评价与后台管理流程一致，界面交互基于 Element Plus 组件，符合常见 Web 使用习惯。配合权限控制与菜单配置，可按岗位分配功能，培训成本较低，推广与日常运维可持续。

3.2 系统功能需求

3.2.1 用户主要功能

面向普通用户，系统需支持服务分类浏览、关键字搜索与推荐、服务人员档案查看、预约下单与在线支付等核心流程，并提供订单状态跟踪、取消或退款申请与售后沟通入口。为形成服务闭环，还需支持服务完成后的评价与打分、浏览记录与收藏管理，以及个人资料、地址与联系方式维护，使用户能在统一平台完成从需求发现到评价反馈的完整业务链路。

3.2.2 管理员主要功能

管理员主要承担平台资源治理与业务审核职责，需要维护服务分类与项目、服务人员资质与排班信息，处理用户预约、支付与退款状态流转，并对评价、投诉与异常订单进行核验与追踪。后台还应支持轮播与公告配置、统计报表与运营数据汇总、角色与菜单权限管理、日志审计与数据导出等功能，以保障订单闭环、规范化运营与跨模块数据一致性。

3.3 非功能性需求分析

3.3.1 性能需求

系统应满足高峰时段多用户并发浏览与下单场景，列表查询、详情加载与搜索筛选需保持稳定的秒级响应；订单提交、支付回调、退款审核等写入操作要确保事务一致性与状态可追溯。为支撑运营统计与历史查询，需通过分页、索引与必要的缓存策略保证报表汇总与多条件查询的性能稳定。

3.3.2 安全性需求

系统涉及用户信息、订单支付与服务人员资质等敏感数据，需基于 Spring Security 与 JWT 实现身份鉴权与接口访问控制，后台通过角色与菜单权限隔离不同操作范围。关键数据传输应采用 HTTPS，支付与退款回调需进行签名校验，登录、订单与退款操作记录审计日志以支持追溯；同时结合参数校验、权限校验与防重复提交机制，确保业务数据安全可靠。

3.3.3 易用性与可维护性需求

系统面向用户、服务人员与管理员三类角色，界面应保持清晰导航与一致交互，常用功能如搜索、下单与评价需在少量步骤内完成，并提供状态提示与异常引导以降低使用成本。前后端采用模块化设计，服务项目、订单、人员与评价等业务模块解耦，统一接口规范与异常处理便于维护；代码层次遵循 Controller/Service/Mapper 分工，方便后续功能扩展与迭代优化。
